# 分析模块协作约定

本文档定义主 agent、各分析子模块、以及后续文档生成 agent 之间的稳定协作契约。

## 1. 开关分层

- `enabledAnalysisModules`
  - 含义：本次报告流程中启用哪些业务分析模块。
  - 作用域：单次请求级别。
  - 示例：`["robotics_risk"]`
- `AGENT_*_ENABLED`
  - 含义：系统是否启用知识图谱、Web 搜索、MCP 等底层能力。
  - 作用域：部署 / 运行时配置级别。
  - 说明：不能把它和报告模块开关混用。

## 2. 主 agent 入参

当前主 agent 已支持以下与分析模块相关的请求字段：

```json
{
  "enabledAnalysisModules": ["robotics_risk"],
  "analysisSharedInputs": {
    "enterpriseName": "石头科技",
    "stockCode": "688169",
    "timeRange": "近30天",
    "reportGoal": "形成机器人行业风险机会简报"
  },
  "analysisModuleInputs": {
    "robotics_risk": {
      "focus": "订单与政策",
      "dimensions": ["政策", "公告", "招中标", "竞争"],
      "sourceControls": {
        "usePolicy": true,
        "useCninfo": true,
        "useBidding": true,
        "preferCache": true,
        "allowLiveFetch": true
      }
    }
  }
}
```

## 3. 共享输入

共享输入由主 agent 在 intake 阶段统一管理，只问一次，供多个已开启模块复用。

| 字段 | 含义 | 是否共享必填 | 说明 |
| --- | --- | --- | --- |
| `enterpriseName` | 企业名称 | 是 | 当前第一优先共享字段 |
| `stockCode` | 股票代码 | 否 | 如有可传，非上市企业允许为空 |
| `timeRange` | 分析时间范围 | 是 | 例如 `近30天`、`近90天` |
| `reportGoal` | 报告目标 | 是 | 例如“识别风险和机会并生成简报” |

规则：

- 共享字段优先于模块专属字段补问。
- 多个模块依赖相同共享字段时，主 agent 只补问一次。
- 未开启模块不得参与共享字段之外的补问逻辑。

## 4. 模块注册契约

每个分析模块通过统一注册表接入，至少声明以下内容：

- `module_id`
- `display_name`
- `shared_fields`
- `module_fields`
- `build_input(shared_inputs, module_inputs, context)`
- `run(payload)`

主 agent 只负责：

1. 根据 `enabledAnalysisModules` 决定要调哪些模块。
2. 先检查共享字段，再检查模块专属字段。
3. 顺序执行已开启模块。
4. 将各模块结果归一化后聚合成统一 handoff bundle。

## 5. 已接入模块

### `robotics_risk`

- 模块名：机器人风险机会洞察
- 当前共享依赖：
  - `enterpriseName`
  - `stockCode`
  - `timeRange`
  - `reportGoal`
- 当前模块专属输入：
  - `focus`（当前必填）
  - `dimensions`（可选）
  - `sourceControls`（可选）

主 agent 会把这些字段构造成 `run_robotics_risk_subagent(...)` 的标准输入，并复用已有：

- `runId`
- `result`
- `documentHandoff`

## 6. 归一化模块输出

各分析模块返回后，主 agent 会整理为统一结构：

```json
{
  "moduleId": "robotics_risk",
  "displayName": "机器人风险机会洞察",
  "status": "done",
  "summary": "政策支持和订单增长构成主要机会。竞争节奏和采购兑现存在波动。",
  "limitations": ["政策样本窗口有限"],
  "runId": "rrisk_parent_001",
  "documentHandoff": {},
  "result": {},
  "targetCompany": {},
  "normalizedInput": {},
  "sourceReferences": [],
  "errorMessage": ""
}
```

字段说明：

- `status`
  - 推荐值：`done` / `partial` / `failed` / `need_input`
- `limitations`
  - 模块级限制说明，主 agent 会继续上抛到聚合 bundle
- `runId`
  - 有运行记录时返回；不是所有模块都强制要求落库
- `documentHandoff`
  - 供文档生成 agent 消费的下游载荷

## 7. 聚合 handoff bundle

主 agent 会生成 `analysisHandoffBundle`，当前结构如下：

```json
{
  "schemaVersion": "analysis_handoff_bundle.v1",
  "enabledModules": ["robotics_risk"],
  "sharedInputSummary": {
    "enterpriseName": "石头科技",
    "stockCode": "688169",
    "timeRange": "近30天",
    "reportGoal": "形成机器人行业风险机会简报"
  },
  "moduleResults": [],
  "moduleRunIds": {
    "robotics_risk": "rrisk_parent_001"
  },
  "moduleSummaries": {
    "robotics_risk": "政策支持和订单增长构成主要机会。竞争节奏和采购兑现存在波动。"
  },
  "documentHandoffs": {
    "robotics_risk": {}
  },
  "unsupportedModules": [],
  "limitations": ["政策样本窗口有限"]
}
```

约束：

- `moduleResults` 必须是归一化后的模块输出集合。
- `documentHandoffs` 供文档生成 agent 直接消费。
- `limitations` 为 bundle 级限制说明，聚合所有模块限制与未注册模块提示。

## 8. 对后续模块开发的要求

后续如接入 `enterprise_operations` 等模块，应遵循同一模式：

1. 在注册表中声明共享字段和模块字段。
2. 提供输入构造函数。
3. 提供统一 `run(...)` 入口。
4. 返回归一化后可被主 agent 聚合的结果。
5. 如需要持久化，自行维护 `runId`，但不得破坏上述输出契约。

## 9. 当前边界

- 当前版本按顺序执行已开启模块，不做并行调度。
- 当前 workspace `/api/workspace/chat` 与 `/api/workspace/chat/stream` 已支持上述分析模块入参。
- 文档生成 agent 尚未实现，但应以 `analysisHandoffBundle` 作为首选输入。
