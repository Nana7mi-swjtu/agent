# 分析模块协作约定

本文档定义主 agent、分析子模块、以及后续文档生成 agent 之间的稳定协作契约。当前版本已经从“共享字段直传”升级为“analysis session + slot-driven intake”。

## 1. 开关分层

- `enabledAnalysisModules`
  - 含义：本次报告流程中启用哪些业务分析模块。
  - 作用域：单次报告会话。
  - 示例：`["robotics_risk"]`
- `AGENT_*_ENABLED`
  - 含义：系统是否启用知识图谱、Web 搜索、MCP 等底层能力。
  - 作用域：部署 / 运行时配置。
  - 说明：不能和报告模块开关混用。

模块开关决定 slot 需求和模块调度，不代表该模块会直接向用户发问。用户侧澄清始终由主 agent 负责。

## 2. 主 agent 与子模块职责

主 agent 负责：

1. 根据 `enabledAnalysisModules` 识别本轮要收集哪些 slot。
2. 恢复或创建当前 `analysis_session`。
3. 把历史兼容入参、用户补充回答统一归一化成 slot 值。
4. 生成当前轮次的补问计划，并只询问缺失 slot。
5. 在 session `ready` 或 `stale` 时顺序调度已开启模块。
6. 聚合模块结果，生成统一 handoff bundle。

分析子模块负责：

1. 声明所需 slot，而不是直接维护面向用户的问答状态。
2. 把 slot 值映射为模块运行时输入。
3. 返回统一归一化结果，供主 agent 聚合。

文档生成 agent 负责：

1. 消费 `analysisHandoffBundle` 和 `documentHandoffs`。
2. 不依赖主 agent 的自然语言回复文本作为主输入。

## 3. Analysis Session

`analysis_session` 是每个 `user_id + workspace_id + role + conversation_id` 唯一对应的一份结构化编排状态。

当前持久化 payload 形状如下：

```json
{
  "sessionId": "asess_xxx",
  "status": "collecting",
  "revision": 1,
  "enabledModules": ["robotics_risk"],
  "slotValues": {},
  "slotStates": {},
  "missingSlots": [],
  "questionPlan": [],
  "moduleStates": {},
  "moduleResults": {},
  "compatibility": {
    "legacySharedInputs": {},
    "legacyModuleInputs": {}
  },
  "handoffBundle": {}
}
```

生命周期状态：

- `collecting`: 仍在补问或等待更多 slot。
- `ready`: 所有必需 slot 已满足，等待模块执行。
- `running`: 已开启模块正在执行。
- `completed`: 当前 enabled modules 都已有最新结果。
- `failed`: 本轮模块执行失败。
- `stale`: 某些结果因 slot 变化失效，需重跑后才重新成为权威结果。

失效规则：

- 共享 slot 变化：所有依赖该 slot 的模块结果标记为 `stale`。
- 模块自有 slot 变化：只标记所属模块结果为 `stale`。
- `revision` 每次 slot 实值发生变化都会递增。

## 4. 全局 Slot Catalog

### 4.1 v1 共享 slot 基线

| slotId | 标签 | scope | valueKind | normalizer | 说明 |
| --- | --- | --- | --- | --- | --- |
| `enterprise_name` | 企业名称 | `shared` | `text` | `enterprise_name` | 需要分析的目标企业 |
| `stock_code` | 股票代码 | `shared` | `text` | `stock_code` | 上市企业 6 位代码，可选 |
| `time_range` | 时间范围 | `shared` | `text` | `time_range` | 如 `近30天`、`近90天` |
| `report_goal` | 报告目标 | `shared` | `text` | `passthrough_text` | 本次分析服务的目标 |
| `analysis_focus_tags` | 分析重点 | `shared` | `multi_choice` | `choice_tags` | 如 政策、公告、招中标、竞争、订单、供应链 |
| `region_scope` | 区域范围 | `shared` | `multi_choice` | `region` | 国家、省份、城市等区域限制 |

### 4.2 v1 基础 valueKind

- `text`
- `single_choice`
- `multi_choice`
- `boolean`

### 4.3 v1 normalizer

- `passthrough_text`
- `enterprise_name`
- `stock_code`
- `time_range`
- `choice_tags`
- `region`

模块可以声明 module-owned slot，但仍需复用相同的基础 `valueKind` 和 normalizer 机制。

## 5. 兼容旧请求载荷

当前主入口仍兼容旧式请求字段：

```json
{
  "enabledAnalysisModules": ["robotics_risk"],
  "analysisSharedInputs": {
    "enterpriseName": "石头科技",
    "stockCode": "688169",
    "timeRange": "近30天",
    "reportGoal": "形成机器人行业风险机会简报"
  },
  "analysisModuleInputs": {
    "robotics_risk": {
      "focus": "订单与政策",
      "dimensions": ["政策", "公告"],
      "sourceControls": {
        "usePolicy": true,
        "useCninfo": true,
        "useBidding": true,
        "preferCache": true,
        "allowLiveFetch": true
      }
    }
  }
}
```

兼容层会先把这些字段映射到 session slot：

- `analysisSharedInputs.enterpriseName` -> `enterprise_name`
- `analysisSharedInputs.stockCode` -> `stock_code`
- `analysisSharedInputs.timeRange` -> `time_range`
- `analysisSharedInputs.reportGoal` -> `report_goal`
- `analysisModuleInputs.robotics_risk.focus` -> `analysis_focus_tags`
- `analysisModuleInputs.robotics_risk.dimensions` -> `robotics_risk.dimensions`

`sourceControls` 当前保留在 `analysis_session.compatibility.legacyModuleInputs`，由模块 adapter 继续消费。

## 6. 模块注册契约

每个分析模块通过统一注册表接入，当前契约字段如下：

- `module_id`
- `display_name`
- `required_slots`
- `optional_slots`
- `slot_definitions`
- `legacy_input_slot_map`
- `legacy_passthrough_fields`
- `slot_mapping(slot_values, compatibility, context)`
- `run(payload)`

约束：

- 主 graph 不应硬编码模块领域字段。
- 模块运行输入必须由 `slot_mapping(...)` 从规范化 slot 值构造。
- 模块输出必须能回写到 session 并参与统一 bundle 聚合。

## 7. 已接入模块：`robotics_risk`

- 模块名：机器人风险机会洞察
- 必需 slot：
  - `enterprise_name`
  - `time_range`
  - `report_goal`
  - `analysis_focus_tags`
- 可选 slot：
  - `stock_code`
  - `robotics_risk.dimensions`

`robotics_risk.dimensions` 是当前模块自有 slot：

| slotId | 标签 | scope | valueKind | normalizer |
| --- | --- | --- | --- | --- |
| `robotics_risk.dimensions` | 分析维度 | `module` | `multi_choice` | `choice_tags` |

主编排不会直接理解机器人领域细节，而是通过该模块的 `slot_mapping(...)` 适配到已有 `run_robotics_risk_subagent(...)` 入口。

## 8. 归一化模块输出

各模块返回后，主 agent 会整理为统一结构：

```json
{
  "moduleId": "robotics_risk",
  "displayName": "机器人风险机会洞察",
  "status": "done",
  "summary": "政策支持和订单增长构成主要机会。竞争节奏和采购兑现存在波动。",
  "limitations": ["公告样本有限"],
  "runId": "rrisk_parent_001",
  "inputSnapshotRevision": 2,
  "documentHandoff": {},
  "result": {},
  "targetCompany": {},
  "normalizedInput": {},
  "sourceReferences": [],
  "errorMessage": ""
}
```

字段要求：

- `status`
  - 推荐值：`done` / `partial` / `failed` / `need_input` / `stale`
- `runId`
  - 有运行记录时返回；不是所有模块都强制要求落库
- `inputSnapshotRevision`
  - 标识该结果对应的 session revision，用于 stale 判断
- `documentHandoff`
  - 提供给文档生成 agent 的模块级下游载荷
- `markdownBody`
  - 如果存在，只是面向聊天展示的编排结果，不是队友模块或正式报告应消费的真源
- `result.readerPacket`
  - 如果模块支持读者向重写，应返回结构化 reader packet，供模块级 LLM renderer、聊天展示和正式报告复用
- `result.factTables`
  - 如果模块支持表格优先展示，应返回标准化 fact tables，保留稳定 `tableId`、列定义、行数据和 trace refs
- `result.chartCandidates`
  - 如果模块支持图表候选，应返回由代码确定性生成的图表候选，显式关联 `sourceTableId`、适用条件和解释边界
- `result.renderedAssets`
  - 如果模块已完成图表渲染，应返回可复用的 rendered assets，保留 `assetId`、`chartId`、`sourceTableId` 和面向读者的说明

## 9. 聚合 Handoff Bundle

主 agent 会生成 `analysisHandoffBundle`，当前 schema 版本为 `analysis_handoff_bundle.v2`：

```json
{
  "schemaVersion": "analysis_handoff_bundle.v2",
  "analysisSession": {
    "sessionId": "asess_xxx",
    "status": "completed",
    "revision": 2
  },
  "enabledModules": ["robotics_risk"],
  "sharedInputSummary": {
    "enterpriseName": "石头科技",
    "stockCode": "688169",
    "timeRange": "近30天",
    "reportGoal": "形成机器人行业风险机会简报",
    "analysisFocusTags": ["政策", "订单"],
    "regionScope": []
  },
  "moduleResults": [],
  "moduleRunIds": {
    "robotics_risk": "rrisk_parent_001"
  },
  "moduleSummaries": {
    "robotics_risk": "政策支持和订单增长构成主要机会。竞争节奏和采购兑现存在波动。"
  },
  "moduleReaderPackets": {
    "robotics_risk": {}
  },
  "moduleTabularArtifacts": {
    "robotics_risk": {
      "factTables": [],
      "chartCandidates": [],
      "renderedAssets": []
    }
  },
  "documentHandoffs": {
    "robotics_risk": {}
  },
  "unsupportedModules": [],
  "staleModules": [],
  "limitations": ["公告样本有限"]
}
```

约束：

- `moduleResults` 必须是归一化后的模块输出集合。
- `moduleReaderPackets` 汇总可直接复用的模块级 reader packet，避免下游重新解析正文。
- `moduleTabularArtifacts` 汇总 fact tables、chart candidates、rendered assets 等结构化展示材料。
- `documentHandoffs` 供文档生成 agent 直接消费。
- `staleModules` 表示当前仍失效、尚未重跑完成的模块。
- `limitations` 聚合所有模块限制和未注册模块提示。

## 10. 对后续模块开发的要求

后续如接入 `enterprise_operations`、`supply_chain_risk` 等模块，应遵循同一模式。当前主链路已经支持通用聚合，新增模块通常不需要修改主 graph，只需要补齐模块自己的注册、运行时输入映射、输出归一化、以及文档 handoff 载荷。

### 10.1 最小接入清单

1. 在 `get_analysis_module_registry()` 中注册 `module_id`、`display_name`、`required_slots`、`optional_slots`。
2. 如有模块自有输入，补充 `slot_definitions`，并复用统一的 `valueKind` / normalizer 机制。
3. 实现 `slot_mapping(slot_values, compatibility, context)`，只从规范化 slot 值和兼容层输入构造运行时 payload。
4. 提供统一 `run(payload)` 入口，避免主 agent 了解模块领域细节。
5. 返回可被 `normalize_analysis_module_output(...)` 接住的结果，至少包含 `status`、`result`、`documentHandoff`、`limitations`，有落库时再补 `runId`。
6. 为该模块实现自己的 `build_document_handoff(...)` 逻辑，不要把文档生成知识硬编码到主 graph。
7. 验证 `analysisHandoffBundle.documentHandoffs.<module_id>` 已能稳定产出，并能被下游文档 agent 直接消费。
8. 为注册、slot 映射、模块输出归一化、handoff bundle 聚合补充测试。

### 10.2 模块级 `documentHandoff` 最低契约

`documentHandoff` 是模块交给下游文档生成 agent 的结构化载荷。它应该是稳定、可序列化、尽量少依赖自然语言上下文的纯数据对象。

建议至少包含以下字段：

- `schemaVersion`
  - 模块自己的 handoff schema 版本，例如 `enterprise_operations_document_handoff.v1`
- `documentType`
  - 文档类型标识，便于下游选择模板或渲染策略
- `recommendedFormat`
  - 推荐输出格式，当前建议为 `markdown`
- `title`
  - 面向最终文档的标题
- `executiveSummary`
  - 供文档 agent 优先使用的摘要对象，不要求固定字段名，但应保持稳定
- `recommendedSections`
  - 按顺序给出建议章节；每个 section 至少应有 `id`、`order`、`title`
- `limitations`
  - 当前模块自己的限制说明

强烈建议补充以下字段，以便文档 agent 在不回看聊天文本的情况下完成首版成稿：

- `targetCompany`
  - 目标企业标识信息
- `analysisScope`
  - 模块分析范围、时间窗、维度等
- `evidenceTable`
  - 证据、引用、定位信息集合
- `evidenceReferences`
  - 面向读者的证据引用列表，应明确标题、来源类型、发布日期、阅读摘要和核验状态
- `citationMap`
  - 章节、事件、证据之间的映射
- `readerPacket`
  - 模块经过主题归并后的 canonical reader payload，建议包含 `executiveSummary`、`opportunities`、`risks`、`evidenceReferences`、`factTableRefs`、`chartCandidateRefs`、`renderedAssetRefs`、`limitations`
- `factTables`
  - 规范化表格集合，应作为聊天展示和正式报告的主要结构化事实来源
- `chartCandidates`
  - 由模块代码从事实表确定性推导出的图表候选，至少应包含 `chartId`、`sourceTableId`、图表类型、caption、解释边界和适用条件
- `renderedAssets`
  - 由模块代码确定性渲染出的可复用图表资产，至少应包含 `assetId`、`chartId`、`sourceTableId`、内容类型和 reader-facing caption

约束：

- `documentHandoff` 应由模块自己生产，主 agent 只负责透传和聚合。
- `documentHandoff` 不承载前端交互状态、轮询进度、临时 UI 字段。
- 下游文档 agent、队友模块和正式报告流程应直接消费 `documentHandoff`、`reportContribution`、`factTables`、`chartCandidates`、`renderedAssets` 等结构化材料，而不是依赖主 agent 的自然语言回复或展示 Markdown。
- 不同模块可以有不同的章节结构和摘要字段，但字段命名在模块内部必须保持稳定。

### 10.3 推荐实现方式

推荐每个模块单独维护自己的 handoff builder，例如：

1. 模块内部先产出领域结果对象。
2. 由 `build_document_handoff(...)` 把领域结果对象转换成稳定 JSON 结构。
3. `run(...)` 最终返回模块结果时，同时附带 `documentHandoff`。

`robotics_risk` 已按此模式实现，可作为参考，但不应被当成唯一 schema 模板。新模块可以复用相同思路，定义自己的 `documentType`、章节结构、证据表和摘要字段。

### 10.4 报告语义模型接入要求

报告生成 agent 面向一类兼容模块工作，而不是为某个具体模块写固定章节。后续模块如果希望参与结构化报告生成，应在模块输出中提供可转换为 `ReportSemanticModel` 的材料，并声明语义贡献能力。

推荐在 `reportContribution` 中补充 `semanticContributionProfile`：

```json
{
  "profileVersion": "report_semantic_contribution.v1",
  "capabilityTypes": ["risk_assessment", "opportunity_assessment", "model_explanation"],
  "supportedSemanticItems": ["finding", "risk_signal", "opportunity_signal", "driver", "evidence", "visual", "recommendation"]
}
```

字段说明：

- `profileVersion`
  - 当前建议使用 `report_semantic_contribution.v1`。
- `capabilityTypes`
  - 描述模块可提供的分析能力类型，例如 `risk_assessment`、`opportunity_assessment`、`financial_health`、`model_explanation`、`source_research`。
- `supportedSemanticItems`
  - 描述模块可稳定提供的语义材料类型，例如 `finding`、`driver`、`evidence`、`visual`、`recommendation`、`limitation`。

`semanticContributionProfile` 只供报告生成器判断兼容性和归一化路径使用，不应出现在最终 Markdown、HTML、预览或下载正文中。

报告可用模块应尽量提供以下材料：

- `findings`
  - 模块已经完成的结论、风险、机会、异常信号或关键判断。
- `evidence`
  - 可解释给用户的来源材料、核验状态、来源类型和支撑关系。
- `attributionInputs`
  - 成因追溯、驱动因素、影响方向、强弱或置信度。
- `logicInputs`
  - 模块内部已经形成的逻辑拆解、推理步骤或约束条件。
- `recommendationInputs`
  - 模块认为可执行的建议、理由、预期收益、风险提示和优先级。
- `modelOutputs`
  - 分数、预测、特征贡献、SHAP、阈值解释等模型输出；必须和现实因果保持边界。
- `visualAssets`
  - 图表、表格、时间线、热力图、模型解释图等可进入报告的视觉材料。
- `factTables`
  - 面向聊天和报告复用的标准化事实表，报告生成器可直接转成 `table_block`。
- `chartCandidates`
  - 从 fact tables 派生的可选图表定义，报告生成器只能从这些候选中选择，不应自由编造图表。
- `renderedAssets`
  - 已渲染好的图表资产，聊天和报告应优先复用；无法直接渲染时再退化到表格或说明文本。
- `limitations`
  - 数据窗口、样本覆盖、模型适用性、来源可靠性等限制。

推荐模块先在代码侧完成：

1. `signal -> theme` 归并
2. `readerPacket` 组装
3. `factTables` 生成
4. `chartCandidates` / `renderedAssets` 生成
5. `evidenceReferences` 归一化

之后如需引入模块级 LLM writer，也应只负责受约束的读者表达，不负责新增事实、数字、来源或图表数据。

约束：

- 模块仍然拥有自己的领域分析过程，报告生成器只负责语义组合、表达、去重、追溯和发布校验。
- 每个实质性结论应能追溯到模块自身的发现、证据、模型输出、图表、逻辑输入或限制说明。
- 模块不应要求最终报告使用自己的 `displayName` 作为章节标题。
- 模块不应要求最终报告展示 `moduleId`、`runId`、`sourceIds`、`traceRefs`、数据库字段或代码字段名。
- 图表和模型输出应提供面向读者的标题、说明、替代文本和解释边界，不能只提供存储路径或内部资产 ID。
- 当模块输出不足、过期、失败或无法核验时，应通过 `status` 和 `limitations` 明确表达，报告生成器会将其降级为限制说明，而不是补写无依据结论。

## 11. 当前边界

- 当前版本按顺序执行已开启模块，不做并行调度。
- 当前 workspace `/api/workspace/chat`、`/api/workspace/chat/stream`、job 路径都支持 analysis session 的恢复与回写。
- 文档生成 agent 尚未实现，但应以 `analysisHandoffBundle` 和 `documentHandoffs` 作为首选输入。
- 当前只有 `robotics_risk` 完成了模块级 `documentHandoff` 落地；其他模块接入时应补自己的 handoff builder 和测试，而不是改写主编排层。
