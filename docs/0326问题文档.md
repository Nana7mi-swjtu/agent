# RAG 已知问题记录（2026-03-26）

## 背景

在当前版本中，用户反馈：聊天一旦触发 RAG，Agent 经常返回：

> 未检索到可支持该问题的证据，请补充资料或换一种问法。

即使问题确实来自已上传文档，也可能出现该现象。

## 主要问题现象

1. 问题内容与文档相关，但返回“无证据”。
2. 前端状态出现 `deterministic (已回退)`。
3. 上传与索引看似成功，但检索命中为空或命中分数被过滤。

## 已确认的根因与风险点

### 1) 嵌入/重排 provider 未正确配置

- 现版本要求显式配置 `RAG_EMBEDDER_PROVIDER` 与 `RAG_RERANKER_PROVIDER`。
- 生产建议：`RAG_EMBEDDER_PROVIDER=openai-compatible`，并配置 `RAG_EMBEDDING_API_KEY` 与 `RAG_EMBEDDING_BASE_URL`。
- 若 provider、模型名或 key 缺失，会在检索/索引阶段报配置错误。

### 2) 语义分块回退到 deterministic

- 当 `chunking` 选择 `semantic_a` / `semantic_b` 时，如分块 API 请求失败、超时、返回格式不合规，会触发回退。
- 回退后会显示 `deterministic (已回退)`。
- 后果：语义分块收益丢失，检索质量进一步下降（尤其配合 deterministic embedder 时）。

排查字段：
- `/api/rag/jobs/{jobId}` 中的：
  - `chunkingApplied.requestedStrategy`
  - `chunkingApplied.strategy`
  - `chunkingApplied.fallbackUsed`
  - `chunkingApplied.fallbackReason`

### 3) 工作区作用域不一致导致“查不到”

- RAG 索引与检索严格按 `user_id + workspace_id` 过滤。
- 如果上传文档时使用的 `workspaceId` 与聊天检索时的 `workspaceId` 不一致，会直接导致命中为空。
- 该类问题表现为：文档存在、索引完成，但当前聊天仍返回无证据。

### 4) 检索阈值过滤导致空结果

- 配置：`RAG_RETRIEVAL_SCORE_THRESHOLD`。
- 当阈值较高（或语义质量本身较弱）时，命中会被过滤掉，最终触发 no evidence。

## 影响范围

- `app/rag/service.py`（检索、阈值过滤、no evidence 文案）
- `app/agent/graph/nodes.py`（RAG 决策与引用包装）
- `app/rag/pipeline/indexer.py`（语义分块与 fallback）
- `frontUI/src/views/app/AppChatView.vue`（策略展示与回退提示）

## 复现最小路径

1. 检查 `RAG_EMBEDDER_PROVIDER` / `RAG_RERANKER_PROVIDER` 是否已显式配置。
2. 上传中文文档并索引。
3. 用“根据文档…”类问题发起聊天。
4. 若 semantic 分块失败回退，或 workspace 不一致，极易出现无证据回复。

## 建议的排查顺序（不涉及代码改造）

1. 先查索引任务结果是否 `done`，并查看 `chunkingApplied.fallbackReason`。
2. 确认上传和聊天使用同一个 `workspaceId`。
3. 临时降低 `RAG_RETRIEVAL_SCORE_THRESHOLD`（如 `0.0` 或更低）验证过滤因素。
4. 抽样调用 `/api/rag/search` 验证是否有 `chunks` 返回。
5. 若长期使用生产能力，评估替换 deterministic embedder（需后续开发）并重建索引。

## 当前结论

“有文档但总是无证据”并非单点故障，通常是以下组合导致：

- deterministic embedder 语义能力不足；
- 语义分块回退；
- workspace 作用域不一致；
- 阈值过滤过严。

以上问题均已定位到可观测字段，可先通过配置和排查路径快速确认具体瓶颈，再决定是否进入下一步代码改造。
