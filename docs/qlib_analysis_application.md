## 2. Qlib 项目概述

### 2.1 项目简介

Qlib 是微软开源的 AI 量化投资平台，核心目标是把量化研究流程标准化，覆盖从数据处理、特征工程、模型训练、预测、回测到结果管理的完整链路。

GitHub: https://github.com/microsoft/qlib

### 2.2 Qlib 的定位

Qlib 不是单一预测模型，而是一个“量化建模与实验框架”。

它强调三件事：

1. 统一数据与特征管道
2. 统一模型训练与评估流程
3. 统一研究结果沉淀和复现

### 2.3 开源与可部署性

1. Qlib 为开源项目，可本地部署
2. 可离线运行，不依赖在线推理服务
3. 支持本地数据目录与离线训练、离线推理

## 3. Qlib 技术能力分析

### 3.1 数据能力

Qlib 支持将行情和特征整理为统一格式，便于模型训练和回测复用。

可接入数据来源包括：

1. Qlib 官方示例数据
2. 公开市场数据源（如 Yahoo Finance）
3. 自有数据源（如 AkShare、TuShare、交易所或内部数据库）

### 3.2 模型能力

Qlib 支持多类模型，常见包括：

1. 传统机器学习模型：LightGBM、XGBoost、线性模型
2. 深度学习模型：LSTM、GRU、Transformer 相关时序模型
3. 排序与打分范式：横截面打分、TopN 选股

在实际工程中，LightGBM 常作为第一版主模型，原因是稳定、训练快、易调参与易解释。

### 3.3 回测与评估能力

Qlib 可对策略效果做历史回测和指标统计，常见指标包括：

1. 年化收益
2. 最大回撤
3. 夏普比率
4. 胜率与换手率

这部分能力对“推荐系统可解释性”很关键，可为每次推荐提供证据链。

## 5. 应用架构设计（建议）

### 5.1 总体架构

建议采用“离线计算 + 在线服务”双层架构：

1. 离线层（Qlib）
- 定时拉取和更新数据
- 训练或增量更新模型
- 生成股票评分、风险评分、解释因子
- 写入 MySQL 推荐结果表

2. 在线层（Flask）
- 查询推荐结果并返回给前端
- 按用户角色调整排序权重
- 将结构化证据注入聊天回答

### 5.2 数据流

1. 数据采集：AkShare 或其他数据源
2. 特征构建：收益、波动、估值、动量等
3. 模型推理：生成每只股票每日预测分
4. 结果落库：推荐表 + 指标表 + 回测摘要表
5. API 输出：TopN 推荐、单票分析、风险提示

### 5.3 角色化策略建议

利用现有角色预设，可做差异化评分权重：

1. 投资者：收益与风险平衡，偏重回报和回撤控制
2. 企业管理者：偏重基本面稳定性与行业景气
3. 监管机构：偏重异常波动、合规风险与提示透明度

可采用融合评分公式：

FinalScore = 0.6 * ModelScore + 0.25 * FactorScore - 0.15 * RiskScore

## 6. 功能落地清单

### 6.1 第一阶段（最小可用）

1. 每日 TopN 推荐接口
- 输入：角色、可选行业、数量
- 输出：股票代码、名称、综合分、推荐理由、风险提示

2. 单票分析详情接口
- 输出：模型分、因子分、风险分、近 30 日表现摘要

3. 推荐解释卡片
- 三条推荐理由
- 两条风险提示
- 一条免责声明

### 6.2 第二阶段（增强）

1. 历史回测报告页
2. 参数化筛选（行业、市值、波动率阈值）
3. 用户自定义偏好（风险等级、持仓周期）

### 6.3 第三阶段（差异化）

1. 深度学习时序模型叠加（LSTM/Transformer）
2. 新闻情绪预训练模型融合
3. 多策略组合和动态再平衡

## 7. 接口设计建议

### 7.1 推荐列表接口

路径建议：/api/workspace/recommendations

请求参数建议：

1. role: investor | enterprise_manager | regulator
2. top_k: 默认 10
3. industry: 可选

响应字段建议：

1. symbol
2. name
3. final_score
4. risk_score
5. reasons
6. warnings
7. as_of_date

### 7.2 单票分析接口

路径建议：/api/workspace/stocks/<symbol>/analysis

响应字段建议：

1. model_score
2. factor_breakdown
3. risk_breakdown
4. backtest_summary
5. recommendation_text

### 7.3 聊天增强接口逻辑

在现有聊天路径中增加“结果检索注入”：

1. 根据用户问题识别股票代码或主题
2. 查询推荐库和分析库
3. 把结构化证据拼接到回答中

这样可以显著提升回答可信度，避免泛化回复。

## 10. 风险与边界

1. 数据质量风险
- 建议增加字段校验和缺失值兜底逻辑

2. 模型漂移风险
- 建议按周或按月滚动重训，监控分布变化

3. 演示稳定性风险
- 必须使用离线结果缓存，避免现场训练

4. 合规风险
- 明确“仅用于研究与教学，不构成投资建议”

---

## 11. 结论

Qlib 对本项目的核心价值在于：

1. 让“股票分析推荐”具备工程化与可复现能力
2. 让你们现有角色化工作台拥有可解释数据证据
3. 以较低改造成本形成“离线智能引擎 + 在线交互服务”的完整架构

建议优先采用“LightGBM 主模型 + 离线每日更新 + 角色化解释输出”的路线，先实现可用、可演示、可扩展的第一版系统。
