# 阶段性说明文档（非参考文档）：RAG 分块当前实现说明

> 文档性质声明：  
> 本文档是**阶段性的实现说明**，用于记录当前代码状态与设计意图，**不是稳定参考文档**。后续实现调整时，本文内容可能过期。

## 1. 当前分块策略

当前仅保留两种分块策略：

- `paragraph`：按段落/文本块拆分（稳定兜底策略）
- `semantic_llm`：由 LLM 理解语义边界并输出语义段（要求原文片段）

当 `semantic_llm` 失败（如模型返回不合法）时，会回退到 `paragraph`。

## 2. `semantic_llm` 的核心流程（内部实现）

代码位置：`app/rag/providers/semantic_chunking_provider.py` 中 `OpenAICompatibleSemanticChunkingProvider`

1. `_build_prompt(...)` 会把输入 `blocks` 组织为带 `block_index` 的样本，并要求模型只返回 JSON。
2. Prompt 明确约束：`segment.text` 必须是 `<exact source span>`，不能总结、改写、扩写。
3. `segment(...)` 构造 OpenAI-compatible 请求，调用 `POST {base_url}/chat/completions`。
4. 解析 `choices[0].message.content` 为 JSON，并提取 `segments`。
5. 对每个 segment 调用 `_align_segment_to_source(...)`：
   - 在原始 block 文本中定位 `segment.text`；
   - 支持直接匹配与空白归一化匹配；
   - 匹配成功后写入 `block_index / offset_start / offset_end`；
   - 匹配失败则抛出 `RAGChunkingError`，上层进入 fallback。
6. 返回 `SemanticSegment` 列表给后续管线做长度约束与入库。

## 3. 文档如何解析为 blocks（对应问题中的第 3 点）

代码位置：`app/rag/pipeline/parsers.py`

- `txt`：整篇文本解析为 1 个 `TextBlock`
- `md`：按标题拆段，每段带 `section`
- `html`：去标签后得到文本 block
- `csv`：按行转 block（`section=row-x`）
- `pdf`：按页转 block（`page`）
- `docx`：按段落转 block（`section=paragraph-x`）

随后在 `normalize_blocks(...)` 中做二次规范化：去空文本、复制 metadata、强制 `source` 存在。

## 4. 如何从 blocks 生成分块（对应问题中的第 4 点）

代码位置：`app/rag/pipeline/indexer.py` + `app/rag/pipeline/chunking.py`

- `paragraph`：
  - 走 `chunker.chunk(...)`；
  - 基于 `chunk_size/overlap` 生成 chunk；
  - 每个 chunk 会记录 `offset_start/offset_end/block_index`。

- `semantic_llm`：
  - 先调用语义 provider 生成语义段；
  - `ensure_semantic_output` 校验结构；
  - `enforce_semantic_bounds` 执行最小/最大 token 约束（必要时合并/拆分）；
  - `semantic_segments_to_payloads` 生成最终 `ChunkPayload`（含 chunk_id、token_count、chunk_strategy 等元数据）。

最终每个 chunk 都会单独向量化并写入向量库（即“一块一个向量”）。

## 5. 关于 txt 文件能否生成多个语句块

可以。

- 解析阶段：`txt` 先得到 1 个 block（整篇文本）。
- 分块阶段：
  - `paragraph` 会按长度切成多个 chunk；
  - `semantic_llm` 会按语义切成多个 segment/chunk。

因此，`txt` 常见结果是：**1 个输入 block -> 多个最终 chunk -> 多个向量**。
