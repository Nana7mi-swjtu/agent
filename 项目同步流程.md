# 项目同步流程

本文档用于在拉取最新代码后，快速完成本地环境、依赖、数据库、前后端联调与 Agent 能力同步。

## 1. 同步后端依赖

本项目推荐使用 `uv`：

```powershell
uv sync
```

若依赖有变更（如 LangGraph/LangChain），必须先执行本步骤再启动服务。

## 2. 同步环境变量（含 Agent / RAG 配置）

以 `.env.example` 为基线补全本地 `.env`。关键项如下：

### 基础配置

- `DATABASE_URL`：必须是 MySQL，格式示例 `mysql+pymysql://user:password@127.0.0.1:3306/app`
- `TEST_DATABASE_URL`：测试库，必须是 MySQL
- `SECRET_KEY`
- `AUTO_CREATE_DB=true`（推荐开发环境开启）

### CORS（前后端分离必配）

- `CORS_ENABLED=true`
- `CORS_ALLOW_CREDENTIALS=true`
- `CORS_ALLOWED_ORIGINS=http://127.0.0.1:4273,http://localhost:4273`
- `CORS_ALLOWED_HEADERS` 需包含 `X-CSRF-Token`

### Agent 运行时配置（新增）

- `AI_PROVIDER`（不限制为 openai）
- `AI_MODEL`
- `AI_API_KEY`
- `AI_BASE_URL`
- `AI_TIMEOUT_SECONDS=30`

注意：模型相关参数仅通过 `.env` 加载，不应硬编码在 Python 代码中。

### RAG / ChromaDB 配置（新增）

- `RAG_ENABLED=false`（默认关闭；开启后启用检索增强）
- `RAG_VECTOR_PROVIDER=chromadb`
- `RAG_EMBEDDER_PROVIDER=deterministic`
- `RAG_EMBEDDING_MODEL=deterministic-v1`
- `RAG_EMBEDDING_VERSION=1`
- `RAG_EMBEDDING_DIMENSION=128`
- `RAG_RETRIEVAL_TOP_K=5`
- `RAG_RETRIEVAL_SCORE_THRESHOLD=0.0`
- `RAG_ALLOWED_FILE_TYPES=pdf,docx,md,txt,html,csv`
- `RAG_UPLOAD_DIR=uploads/rag`
- `RAG_AUTO_INDEX_ON_UPLOAD=true`
- `RAG_CHUNK_SIZE=1200`
- `RAG_CHUNK_OVERLAP=150`
- `RAG_INDEX_MAX_WORKERS=2`
- `RAG_CHROMADB_PERSIST_DIR=uploads/chromadb`
- `RAG_CHROMADB_COLLECTION_PREFIX=rag`

RAG 详细运维说明见：`docs/ChromaDB.md`。

## 3. 同步数据库

应用启动时会自动执行：

1. 数据库存在性检查/创建（MySQL）
2. ORM `create_all`
3. `users` 结构补齐（`nickname/avatar_url/preferences/updated_at`）

如需手动初始化，可按 `migrations/` 顺序执行：

1. `000_create_databases.sql`
2. `001_create_tables.sql`
3. `002_alter_users_profile_fields.sql`
4. `003_add_preferences_to_users.sql`
5. `004_add_rag_tables.sql`（RAG 文档/分片/索引任务/查询日志）

## 4. 启动后端并做基础健康检查

```powershell
uv run main.py
```

默认端口 `8000`，健康检查：

- `GET /health` 返回 `{"ok": true}`

## 5. 启动前端并对齐 API 基址

在 `frontUI/.env.development` 中确认：

- `VITE_API_BASE_URL=http://127.0.0.1:8000`

启动前端：

```powershell
cd frontUI
npm install
npm run dev
```

访问：`http://127.0.0.1:4273/#/login`

## 6. 联调关键链路（重点）

按下面顺序验证可快速定位问题：

1. 登录成功，浏览器携带会话 Cookie
2. 进入工作台，选择角色（会写入 `users.preferences.workspace.role`）
3. 进入聊天页发送消息
4. 后端返回：
   - 成功：`200`，包含 `data.role` / `data.systemPrompt` / `data.reply`
   - Agent 配置缺失或运行失败：`502` + `{"ok": false, "error": "agent service unavailable"}`

若聊天提示 `please select a role first`，优先检查角色选择请求是否成功及数据库 `preferences.workspace.role` 是否已持久化。

### RAG 链路（当 `RAG_ENABLED=true`）

1. 上传文档：`POST /api/rag/upload`（`multipart/form-data`，字段 `file`，可选 `workspaceId`）
2. 触发索引：`POST /api/rag/index`（`workspaceId` + `documentId`）
3. 查询任务状态：`GET /api/rag/jobs/<job_id>?workspaceId=...`
4. 检索验证：`POST /api/rag/search`（`query` / `topK` / `filters`）

说明：
- 所有写请求同样需要登录会话 + `X-CSRF-Token`
- 当 `RAG_ENABLED=false` 时，`/api/rag/*` 返回 `404`（`rag feature is disabled`）

## 7. 同步后回归测试

```powershell
uv run pytest -q
```

若仅验证 workspace/agent 相关改动，可先定向跑 `tests/test_flows.py` 中对应用例。

## 8. 常见问题排查

1. `DATABASE_URL must use MySQL`  
   检查 `DATABASE_URL` 是否为 `mysql+pymysql://...`。

2. 测试被跳过  
   检查 `TEST_DATABASE_URL` 是否设置且为 MySQL。

3. 认证写接口返回 `403 csrf token missing or invalid`  
   登录后请求头附带 `X-CSRF-Token`，并确保前端 `credentials: include`。

4. 聊天返回 `agent service unavailable`  
   检查 `AI_MODEL`、`AI_API_KEY`、`AI_BASE_URL` 是否有效；查看后端日志定位 provider 调用失败。

5. 角色已选但聊天仍提示 `please select a role first`  
   检查 `/api/workspace/context` `PATCH` 是否成功，以及 `users.preferences.workspace.role` 是否写入。

6. RAG 接口返回 `404 rag feature is disabled`  
   检查 `.env` 中 `RAG_ENABLED` 是否为 `true`，并重启后端服务。

7. 文档上传报“不支持格式”  
   检查 `RAG_ALLOWED_FILE_TYPES` 是否包含对应后缀（如 `pdf,docx,md,txt,html,csv`）。

8. 检索没有命中  
   先确认索引任务状态为 `done`；再检查 `workspaceId` 一致性、`RAG_RETRIEVAL_SCORE_THRESHOLD` 是否过高。

## 9. 推荐同步顺序（标准）

```powershell
uv sync
uv run main.py
cd frontUI
npm install
npm run dev
```

最后执行：

```powershell
cd ..
uv run pytest -q
```

若包含 RAG 联调，请补充执行：

```powershell
# 以已登录态调用 /api/rag/upload -> /api/rag/index -> /api/rag/jobs -> /api/rag/search
```
